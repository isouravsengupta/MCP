# SPI MCP Rebuild and Recover Runbook

Use this runbook to recreate SPI MCP in a new AWS account, sandbox, or when local state is lost.

## 1) Source of truth

- Repository: this `spi-mcp` folder (commit/push to remote regularly)
- Context repos:
  - `business_logic`
  - `airflow-dags-uip-gdso`
- Backup archive (optional): generated by `ops/backup-mcp.sh`

## 2) Local bootstrap

```bash
npm install
npm run build
```

## 3) Environment variables

Minimum:

```bash
export AWS_REGION=us-west-2
export MCP_AUTH_TOKEN="<secure-random-token>"
```

For current working auth (recommended in Lambda):

```bash
export SPI_SNOWFLAKE_AUTH_MODE=password
export SNOWFLAKE_PASSWORD="<snowflake-service-password>"
```

For SSO wiring test:

```bash
export SPI_SNOWFLAKE_AUTH_MODE=sso
export SNOWFLAKE_AUTHENTICATOR=externalbrowser
```

## 4) Deploy

```bash
./infra/deploy-prod.sh
```

## 5) Post-deploy checks

1. `initialize` works on endpoint.
2. `tools/list` returns tools.
3. `snowflake_connectivity_probe` succeeds.
4. `validate_metric_regressions` returns `ok: true` for core checks.
5. Or run all checks in one command:

```bash
export MCP_ENDPOINT="https://<api-id>.execute-api.<region>.amazonaws.com/prod/"
./ops/smoke-check.sh
```
6. Slack MCP config uses endpoint URL with trailing slash:
   - `https://<api-id>.execute-api.<region>.amazonaws.com/prod/`

## 7) Recovery checklist for production migration

- Create persistent AWS account resources (VPC, NAT/EIP, IAM role boundaries).
- Move secrets to production AWS Secrets Manager.
- Set `SPI_DISABLE_AUTH` to `false` once Slack auth model is finalized.
- Keep service account mode as fallback while SSO mode is validated.
- Version-control `resources/metric_mappings.json` and `resources/column_dictionary.json`.
- Add regression tests for top dashboard metrics before cutover.
